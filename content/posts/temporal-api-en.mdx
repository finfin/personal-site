On March 11, 2026, Temporal reached TC39 Stage 4 and became part of the ES2026 specification. It is now natively shipped in Chrome 144+, Firefox 139+, and Edge 144+.

Temporal is JavaScript's new datetime API — the language-level answer to thirty years of `Date` design problems. If you've been reaching for `dayjs`, `date-fns`, or `moment.js` to fill gaps that `Date` leaves open, Temporal is what makes those libraries unnecessary.

## Temporal's Types

Temporal lives as a top-level namespace (like `Math` or `Intl`) with a family of distinct types, each purpose-built for a specific use case — unlike `Date`, which tries to be everything at once:

| Type | When to use |
|------|-------------|
| `Temporal.PlainDate` | Birthdays, holidays — date only, no time, no timezone |
| `Temporal.PlainDateTime` | Calendar events in a "local" context |
| `Temporal.ZonedDateTime` | Meetings, flights — explicit IANA timezone + DST handling |
| `Temporal.Instant` | An exact point in time, for storage or cross-timezone comparison |
| `Temporal.Duration` | The span between two points in time |
| `Temporal.PlainTime` | Time-of-day without a date |

All Temporal objects are **immutable** — operations return new objects, originals are never modified. When in doubt, start with `ZonedDateTime`.

## Example 1: ZonedDateTime Handles DST Automatically

This is one of the most impressive parts of Temporal. During a DST transition, some moments simply don't exist — `Date` leaves that problem to you, `Temporal.ZonedDateTime` handles it automatically.

On March 29, 2026, the UK clocks spring forward from 01:00 to 02:00 — meaning `01:30` doesn't exist:

```javascript
// London DST starts: 2026-03-29 01:00 -> 02:00
const zdt = Temporal.ZonedDateTime.from(
  '2026-03-29T00:30:00+00:00[Europe/London]'
);

const plus1h = zdt.add({ hours: 1 });
console.log(plus1h.toString());
// "2026-03-29T02:30:00+01:00[Europe/London]"
// 01:30 doesn't exist — Temporal skips straight to 02:30
```

## Example 2: Instant for Cross-Timezone Display

`Temporal.Instant` represents an exact point in time with no timezone or calendar semantics — just nanoseconds since the Unix epoch. Store it, transmit it, then convert for each user's timezone:

```javascript
const instant = Temporal.Instant.from('2026-02-25T15:15:00Z');

instant.toZonedDateTimeISO('Europe/London').toString();
// "2026-02-25T15:15:00+00:00[Europe/London]"

instant.toZonedDateTimeISO('America/New_York').toString();
// "2026-02-25T10:15:00-05:00[America/New_York]"
```

## Example 3: Duration Arithmetic

With `Date`, duration calculation means subtracting milliseconds and manually converting to meaningful units.

**Before (Date):**

```javascript
function getDuration(start, end) {
  const diffMs = new Date(end) - new Date(start);
  return {
    days: Math.floor(diffMs / (1000 * 60 * 60 * 24)),
    hours: Math.floor((diffMs % (1000 * 60 * 60 * 24)) / (1000 * 60 * 60)),
    minutes: Math.floor((diffMs % (1000 * 60 * 60)) / (1000 * 60)),
  };
}
```

**After (Temporal):**

```javascript
function getDuration(startStr, endStr) {
  const start = Temporal.PlainDateTime.from(startStr);
  const end = Temporal.PlainDateTime.from(endStr);

  const duration = start.until(end, { largestUnit: 'day' });
  return {
    iso: duration.toString(),   // "P1DT8H30M"
    days: duration.days,
    hours: duration.hours,
    minutes: duration.minutes,
    totalHours: duration.total('hours'),
  };
}
```

`start.until(end)` returns a `Temporal.Duration` object. The ISO 8601 representation (`P1DT8H30M`) is human-readable, and `.total('hours')` converts the entire duration to a single unit.

## Example 4: Calendar Support

Temporal natively supports non-Gregorian calendars. "Add one month" operates in the rules of the specified calendar, not Gregorian-then-reformat:

```javascript
const today = Temporal.PlainDate.from('2026-03-11[u-ca=hebrew]');
today.toLocaleString('en', { calendar: 'hebrew' });
// '22 Adar 5786'

const nextMonth = today.add({ months: 1 });
nextMonth.toLocaleString('en', { calendar: 'hebrew' });
// '22 Nisan 5786'
```

With `Date`, adding a Gregorian month and then displaying in Hebrew gives the wrong day — the calendar systems don't have the same month lengths.

## Why Did Date Need Replacing?

`Date` was directly copied from Java's buggy 1995 implementation. Three fundamental design problems accumulated over the decades, driving the creation of moment.js, dayjs, and date-fns — together exceeding 100 million weekly downloads by 2026.

### Mutation hides inside function calls

`Date` is mutable. Passing a `Date` into a function can silently modify the original, without the caller knowing:

```javascript
const date = new Date('2026-02-25T00:00:00Z');

function addOneDay(d) {
  d.setDate(d.getDate() + 1); // modifying the argument directly
  return d;
}

addOneDay(date);
console.log(date.toISOString()); // "2026-02-26T00:00:00.000Z" — mutated!
```

`addOneDay` isn't buggy on its own — `Date` is inherently mutable, so any function receiving one can change it. The caller has no way to know. Temporal objects are always immutable, so this problem doesn't exist.

<JSPlayground
  title="Try it: addOneDay mutation bug"
  defaultCode={`const date = new Date('2026-02-25T00:00:00Z');

function addOneDay(d) {
  // looks harmless, but modifies the original
  d.setDate(d.getDate() + 1);
  return d;
}

console.log('before:', date.toISOString().slice(0, 10));
addOneDay(date);
console.log('after: ', date.toISOString().slice(0, 10)); // silently mutated

// safe version requires manual cloning at call site
const date2 = new Date('2026-02-25T00:00:00Z');
function addOneDaySafe(d) {
  const copy = new Date(d);
  copy.setDate(copy.getDate() + 1);
  return copy;
}
addOneDaySafe(date2);
console.log('safe:', date2.toISOString().slice(0, 10)); // unchanged`}
/>

### Month arithmetic silently overflows

`Date` doesn't clamp invalid dates — it rolls overflow into the next month:

```javascript
const billingDate = new Date('Jan 31 2026');
billingDate.setMonth(billingDate.getMonth() + 1);
// Expected: Feb 28    Actual: Mar 02
```

Temporal automatically clamps results to the last valid day of the month, and throws a `RangeError` for invalid dates rather than silently overflowing.

### Parsing behavior is inconsistent across browsers

The same string can produce different results depending on the browser:

```javascript
new Date('2026-06-25 15:15:00').toISOString();
// Chrome: local timezone   Firefox: might throw   Safari: UTC
```

This string looks like ISO 8601 but isn't quite — the spec historically left "almost-ISO" parsing behavior undefined. Temporal only accepts strictly defined formats and throws on anything ambiguous.

## Gotchas Worth Knowing

### 1. Date comparison requires compare()

```javascript
// ❌ Won't work — Temporal objects aren't primitives
date1 > date2

// ✅ Correct
Temporal.PlainDate.compare(date1, date2) // Returns -1, 0, or 1
```

### 2. dayOfWeek is ISO 8601 (1 = Monday)

`Date.getDay()` returns `0` for Sunday and `6` for Saturday. Temporal uses ISO 8601: `1` is Monday, `7` is Sunday. Don't port old code assuming the old convention.

### 3. largestUnit matters for until()

```javascript
const start = Temporal.PlainDate.from('2026-01-15');
const end = Temporal.PlainDate.from('2026-02-15');

start.until(end, { largestUnit: 'day' })   // P31D
start.until(end, { largestUnit: 'month' }) // P1M
```

The two results are different representations of the same duration, but the choice affects downstream arithmetic.

### 5. Pick the right type

- `PlainDate` for birthdays, holidays, deadlines — no time component
- `ZonedDateTime` for meetings, flights — explicit timezone + DST awareness
- `Instant` for stored timestamps and cross-timezone comparisons
- When in doubt, start with `ZonedDateTime`

### 6. Browser support

Temporal became part of ES2026 in March 2026 and is natively shipped in modern browsers:

- **Chrome 144+** (since January 2026)
- **Firefox 139+** (since May 2025)
- **Edge 144+** (since January 2026)
- **Safari** (partial support in Tech Preview)

If you need to support older browsers, use `@js-temporal/polyfill`:

```bash
npm install @js-temporal/polyfill
```

```javascript
import { Temporal } from '@js-temporal/polyfill';
```

## The Summary

| Problem | Date | Temporal |
|---------|------|----------|
| Mutability | Functions can silently modify the object they receive | All operations return new objects |
| Month overflow | Silently rolls into the next month | Clamps to the last valid day |
| String parsing | Inconsistent across browsers | Strict, well-defined ISO 8601 |
| Timezone support | UTC and local only | Full IANA timezone support |
| DST handling | Manual | ZonedDateTime handles it automatically |
| Invalid dates | Silent overflow | Throws a RangeError |
| Duration type | Milliseconds only | Rich Duration with `total()` and unit conversion |
| Calendar support | None | Hebrew, Japanese, Islamic, and more |
| Comparison | `>`, `<` (unreliable) | `Temporal.*.compare()` |

After nearly thirty years, JavaScript finally has a datetime API that doesn't require a library to use correctly.
